# 部室予約システム(Firebase + Googleログイン版)

軽音楽部の部室予約Webアプリです。GitHub Pagesなどの静的ホスティングで公開し、
データの保存にはFirebase(無料枠)、ログインには学校のGoogleアカウントを使います。

**主な機能**
- 学校のGoogleアカウントでのログイン(サイト全体がログイン必須)
- バンド・個人練習の登録(パスワードなし)と、それぞれの予約・取消
- スプレッドシート風の予約カレンダー(30分単位・当日から2週間先まで)
- 連絡事項(注意事項・ホワイトボード優先順位・部会日程・活動禁止日・その他バンド一覧・今後のライブ予定)
- 備品持ち出し記録
- 各種リンク集

> このREADMEの手順どおりに進めれば、GitHubの操作に慣れている方なら
> 30分〜1時間程度で公開できるはずです。

---

## 0. ファイル構成

```
index.html            # 読み込み専用のごく薄いシェル(編集不要)
firebase-config.js     # ★あなたが編集する唯一のファイル(Firebaseの接続情報・学校ドメイン)
firebase-adapter.js    # Firebase SDKの読み込み・ラップ(編集不要)
app.js                 # アプリ本体のロジック(編集不要)
firestore.rules        # Firestoreのセキュリティルール(Firebaseコンソールに貼る、ドメイン部分だけ編集)
test/                  # 動作確認用のテストコード(公開サイトには不要、削除してよい)
```

公開に必要なのは `index.html` `firebase-config.js` `firebase-adapter.js` `app.js` の4つです。

---

## 1. Firebaseプロジェクトを作る

1. https://console.firebase.google.com/ を開き、Googleアカウントでログイン。
2. 「プロジェクトを追加」→ 名前を入力(例: `bandroom-booking`)→ Google アナリティクスは不要なのでオフでよい → 作成。

## 2. Authentication(認証)でGoogleログインを有効化する

1. 左メニュー「Authentication」→「始める」。
2. 「Sign-in method」タブ →「Google」を選択 →「有効にする」。
3. プロジェクトのサポートメールを選択(自分のGoogleアカウントでよい)→ 保存。
   *(これを忘れると、ログインボタンを押してもエラーになります)*

## 3. Firestore Database を作る

1. 左メニュー「Firestore Database」→「データベースの作成」。
2. ロケーションは `asia-northeast1`(東京)など、日本に近いものを選択。
3. モードは「本番環境モード」を選択(下の手順6で正式なルールを設定するため)。

## 4. ウェブアプリを登録し、設定値を取得する

1. Firebaseコンソールのトップ(プロジェクトの概要)→ `</>` (ウェブ)のアイコンをクリック。
2. アプリ名を入力(例: `bandroom-web`)→ Firebase Hostingの設定はスキップしてよい → 登録。
3. 表示される `firebaseConfig = { apiKey: ..., authDomain: ..., ... }` の中身をコピー。
4. このリポジトリの `firebase-config.js` を開き、`firebaseConfig` の値を、コピーした値に置き換える。

## 5. 学校のドメインを設定する(★重要)

`firebase-config.js` の一番下にある

```js
export const requiredEmailDomain = "YOUR_SCHOOL_DOMAIN";
```

の `YOUR_SCHOOL_DOMAIN` を、実際の学校のGoogle Workspaceドメインに書き換えます(例: `xxx.ac.jp`)。

```js
export const requiredEmailDomain = "xxx.ac.jp";
```

この値は「ログイン時にアカウント選択画面をその学校のアカウントに絞り込むヒント」として使われます。
**実際のアクセス制御(このドメイン以外のアカウントを弾く)は、次の手順6のFirestoreルール側で行います。**

## 6. セキュリティルールを設定する(★重要)

1. このリポジトリの `firestore.rules` を開き、ファイル内にある `YOUR_SCHOOL_DOMAIN` という文字列(1箇所)を、
   手順5と**同じ学校のドメイン**に書き換える。

   ```
   // 変更前
   && request.auth.token.email.lower().matches('.*@YOUR_SCHOOL_DOMAIN$');

   // 変更後の例
   && request.auth.token.email.lower().matches('.*@xxx.ac.jp$');
   ```

2. Firebaseコンソール →「Firestore Database」→「ルール」タブを開く。
3. 書き換えた `firestore.rules` の内容を**全部コピー**して、エディタの内容を丸ごと置き換える。
4. 「公開」をクリック。

⚠️ **この`YOUR_SCHOOL_DOMAIN`を書き換え忘れると、誰もログインできなくなります**(安全側に倒れて全員拒否になる仕組みです)。動かない場合はまずここを疑ってください。

このルールにより、以下がFirebase側(サーバー側)で強制されます。

- **学校ドメインのGoogleアカウントでログインしていない人は、閲覧を含めて何もできません。**
- ログインさえしていれば、バンド・個人の登録/削除、予約/取消、連絡事項・備品持ち出し記録・各種リンクの編集は、誰でも行えます(部員内の信頼にもとづく運用です)。
- 予約は30分単位・最大2時間までという制約もここで強制されます。

*(「2週間で最大2件」「時間帯の重複禁止」「営業時間(祝日判定含む)」は複数の予約をまたいで判定する必要があり、Firestoreルールだけでは効率的に検証できないため、引き続きアプリ側(app.js)でチェックしています。学校ドメインのアカウントを持つ人が、自分名義でこれらの制限をブラウザの開発者ツールなどですり抜けることは技術的には可能です。)*

## 7. Authorized domains(承認済みドメイン)にGitHub Pagesのドメインを追加する

Googleログインのポップアップは、Firebaseが許可したドメインからしか開始できません。

1. Firebaseコンソール →「Authentication」→「Settings」タブ →「承認済みドメイン」。
2. 「ドメインを追加」から、GitHub PagesのURLのドメイン部分(例: `hajime-18.github.io`)を追加。

これを忘れると、「Googleでログイン」ボタンを押した際にエラーになります。

## 8. GitHubリポジトリにpushして、GitHub Pagesで公開する

```bash
git add -A
git commit -m "Googleログイン方式に変更"
git push
```

その後、GitHubのリポジトリページで「Settings」→「Pages」から公開設定を確認してください
(`Branch: main` / `Folder: / (root)`)。数十秒〜数分後、公開URLでアクセスできます。

## 9. 本番投入前に必ず確認すること

100人に展開する前に、**最低でも2〜3人の実際の部員に試してもらってください**。

- [ ] 学校ドメインのGoogleアカウントでログインでき、アプリが表示される
- [ ] 学校ドメイン以外のGoogleアカウント(個人のGmailなど)でログインすると、利用できない旨の画面が出る
- [ ] 別の端末でバンド登録・予約すると、自分の画面にもリアルタイムで反映される
- [ ] 個人練習としても登録・予約ができる
- [ ] 連絡事項・備品持ち出し記録・各種リンクが、ログインさえしていれば編集・反映される

## 10. 運用上の注意(正直な限界)

- **本当に強制されるのは**: 学校ドメインのGoogleアカウントでログインしていない人は、閲覧も編集も一切できない、という部分です。
- **強制されていないのは**: ログインした人同士の間での制限です。「2週間で最大2件」「時間帯の重複」「祝日の営業時間」といった業務ルールはブラウザ側のコードでチェックしているため、技術に詳しい部員が開発者ツールなどを使えばすり抜けられる可能性があります。また、他人の予約や登録を削除することも(ログインさえしていれば)可能です。これは「学校ドメインの人だけが対象で、あとは信頼で運用する」という設計判断にもとづくものです。
- Firestore・Authenticationとも無料枠(Sparkプラン)で、部活の利用規模(100人・1日数件の予約)であれば費用が発生することはまずありません。
